# Twilight Imperium hero cards and versioned support cards

## Executive summary

In official **Twilight Imperium Fourth Edition** product history, **hero leader cards do not exist in the base game**. Leaders are introduced by **Prophecy of Kings**, which adds leaders for the original base-game factions plus seven new factions, and it also replaces the base-game **Diplomacy** and **Construction** strategy cards with revised versions. **Codex III** adds the **Council Keleres** faction and introduces the first published **leader ╬® replacements** for existing factions. **ThunderŌĆÖs Edge** then physically compiles prior Codex gameplay content, adds five new factions, and replaces **Construction** and **Warfare** with new revised versions. ## Scope and method

I used a strict inclusion rule for the ŌĆ£comprehensiveŌĆØ tables below. The report includes:

- every **hero leader card** I could verify from the public card-transcription corpus and product documents,
- every **versioned support card** that materially changes hero play or the hero timing environment,
- a compact **promissory-note inventory** because hero effects such as **The Oracle** and several commander/agenda interactions can make note versioning matter,
- and a short list of **action/agenda cards** whose text directly changes **strategy selection** or **strategic action resolution**, because those are the main non-leader windows that repeatedly intersect hero timing. ŅłĆciteŅłéturn20view0Ņłéturn22view0Ņłéturn25view0Ņłéturn37view0Ņłü

Where public official card images were easily available, I prioritized them. Where they were not easily indexed, I used a combination of **official set/rules pages** for set membership and replacement status, together with the **Twilight Imperium Wiki/Fandom card indexes** for card-by-card wording. Because of that limitation, the tables below use **authoritative summaries** rather than reproducing long verbatim card text. That is the safer and more reliable way to normalize multiple printings into database-ready records. ## Recommended database schema

For a site that needs to survive card revisions, optional Codex adoption, and future expansion waves, the best pattern is **cards as identities** and **printings/versions as children**. A hero such as **Dannel of the Tenth** should be one logical card identity with multiple version rows if you want users to compare **Spinner Overdrive** against **Quantum Dissemination ╬®**. Likewise, **Construction** should exist once as a card identity with separate version rows for **base**, **PoK revision**, and **ThunderŌĆÖs Edge revision**.

A practical relational schema looks like this:

```mermaid
erDiagram
    SETS {
        uuid id PK
        string name
        string set_type
        date release_date
        boolean official
    }

    CARD_FAMILIES {
        uuid id PK
        string canonical_name
        string canonical_type
        string faction_name
        boolean generic_card
        text design_notes
    }

    CARD_VERSIONS {
        uuid id PK
        uuid card_family_id FK
        uuid set_id FK
        string printed_name
        string subtype
        string version_tag
        string timing_window
        text rules_summary
        text exact_text_optional
        boolean current_official_standard
        boolean optional_print_and_play
        boolean ambiguous_fields
    }

    REPLACEMENTS {
        uuid id PK
        uuid newer_version_id FK
        uuid older_version_id FK
        string replacement_kind
        text rationale
    }

    VALIDITY_PROFILES {
        uuid id PK
        string name
        boolean uses_pok
        boolean uses_codex
        boolean uses_thunders_edge
        text notes
    }

    VERSION_VALIDITY {
        uuid id PK
        uuid card_version_id FK
        uuid validity_profile_id FK
        boolean legal
        text note
    }

    SOURCES {
        uuid id PK
        uuid card_version_id FK
        string source_kind
        string citation_label
        text source_note
    }

    SETS ||--o{ CARD_VERSIONS : contains
    CARD_FAMILIES ||--o{ CARD_VERSIONS : has
    CARD_VERSIONS ||--o{ SOURCES : documented_by
    CARD_VERSIONS ||--o{ VERSION_VALIDITY : legal_in
    VALIDITY_PROFILES ||--o{ VERSION_VALIDITY : defines
    CARD_VERSIONS ||--o{ REPLACEMENTS : newer
    CARD_VERSIONS ||--o{ REPLACEMENTS : older
```

Recommended field types:

- `canonical_type`: enum such as `hero_leader`, `strategy`, `action`, `agenda`, `promissory`
- `version_tag`: enum such as `original`, `omega`, `revised`, `replacement`
- `timing_window`: short text, not a hard enum, because TI timing language is varied
- `rules_summary`: long text
- `exact_text_optional`: long text, nullable
- `current_official_standard`: boolean
- `optional_print_and_play`: boolean
- `ambiguous_fields`: boolean
- `replacement_kind`: enum such as `codex_omega`, `expansion_revision`, `physical_reprint`

This model makes the hardest cases easy:

- **Xxcha hero**: one family, two versions.
- **Yin hero**: one family, two versions.
- **Construction**: one family, three versions.
- **Keleres hero**: either one family with three mutually exclusive branch versions, or three separate card families under faction `Council Keleres`, depending on how you want search to behave.
- **Ambiguous TE hero names**: version exists now with `ambiguous_fields = true`, later patched when an indexed official/public card image is available.
