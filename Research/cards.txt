# Twilight Imperium Card Version Mapping Report

## Executive summary

The cleanest way to track *Twilight Imperium Fourth Edition* card provenance is to treat **Base Game**, **Prophecy of Kings**, **Codex I–IV**, and **Thunder’s Edge** as distinct source layers, then track **replacement chains** separately from **new-addition chains**. Official Fantasy Flight support materials clearly show the publication history of the relevant documents: the base rules, the *Prophecy of Kings* rulebook, all four Codex volumes and card packs, the early standalone updated Diplomacy and Technology files, the Codex 4.5 technology update file, and the *Thunder’s Edge* rulebook. citeturn11search0turn12search0turn8view0

For the categories you asked about, the version story is:

* **Strategy cards:** Base has the original eight. **Diplomacy** is officially updated in the early 2020 “Alternate Diplomacy” release and later rolled into Codex I-era materials; **Construction** gets a wording update in *Prophecy of Kings* and a functional replacement again in *Thunder’s Edge*; **Warfare** gets a *Thunder’s Edge* updated version. citeturn10view5turn21view4turn14view2turn24search9
* **Action cards:** Base has the original action deck; **Codex I** adds 20 cards; **PoK** adds 20 more cards; **Thunder’s Edge** adds 20 more cards and republishes Codex action cards with several wording updates. Community card indexes and the TI4 JSON data set are consistent on the Base / Codex I / PoK / Thunder’s Edge additions, but there is one important dispute: one community JSON data set tags several updated action cards as **Codex III**, while the TI Wiki’s version tracking and *Thunder’s Edge* section attribute those wording updates to **Thunder’s Edge**. I treat those as **Thunder’s Edge** updates, and I flag the discrepancy explicitly below. citeturn26view0turn26view1turn26view2turn26view3turn27view0turn23view0turn24search1
* **Political / agenda cards:** Base contains 50 agendas. When using *Prophecy of Kings*, **13 base agendas are removed**, **13 new agendas are added**, and **Representative Government** is replaced by a new card of the same name with different wording. citeturn14view1turn34view0
* **Technology cards:** The most important official replacement chain is **Magen Defense Grid → Magen Defense Grid Ω → Magen Defense Grid ΩΩ** and **X-89 Bacterial Weapon → X-89 Bacterial Weapon Ω → X-89 Bacterial Weapon ΩΩ**. Codex I also updates **Magmus Reactor** and **Yin Spinner**. *Prophecy of Kings* adds eight generic technologies and many faction technologies. Codex III introduces **I.I.H.Q. Modernization** for Council Keleres; the TI Wiki further notes that this card is moved to a **Breakthrough** in *Thunder’s Edge*, which makes it a genuine version-tracking edge case. citeturn41view0turn41view1turn30view3turn41view2turn37view2

Because Fantasy Flight’s current support pages do not expose a clean official per-card browser for every base and PoK card face, this report uses **official FFG support documents first** for release provenance, then cross-checks per-card names and wording against the well-maintained TI Wiki and the `ti4-mcp` JSON data set. Where those community sources disagree or overstate certainty, I mark the case as **ambiguous**. citeturn11search0turn38view0turn15view0

## Source hierarchy and method

I used the following source hierarchy.

**Official sources** came first: Fantasy Flight’s *Twilight Imperium Fourth Edition* support page and linked files, including the base rules documents, the *Prophecy of Kings* rulebook, *Twilight Codex* Volumes I–IV and their card PDFs, the early standalone “Alternate Diplomacy” and “Alternate Technology Cards” files, the “Codex 4.5 Technology Card Updates” file, and the *Thunder’s Edge* rulebook. The official support page itself is important evidence because it establishes which documents exist, when Fantasy Flight dated them, and which update files are meant to be used with the game. citeturn11search0turn12search0turn8view0

**Primary community verification** came from the TI Wiki pages for Action Cards, Agenda Cards, Strategy Cards, Technology, the color-specific technology pages, Faction Technologies, Omegas, and Version Tracking. Those pages are especially useful because they label **expansion symbols**, **Ω / ΩΩ chains**, and **removed-by-PoK** agenda status in human-readable form. citeturn14view0turn14view1turn14view2turn14view3turn30view0turn30view1turn30view2turn30view3turn38view0

**Machine-readable community verification** came from the `surewht/ti4-mcp` repository, whose README explains the data model and explicitly says that when codex volumes are included, updated components automatically replace older Base / PoK versions. Its JSON files for action cards, agendas, strategy cards, and technologies were useful for checking source tags and replacement chains. I did **not** treat that repository as authoritative when it conflicted with the TI Wiki’s version tracking or the official document structure. citeturn15view0turn20view0turn20view2turn20view3turn20view4

For this report, I interpret **Current official status** as a practical usage label:

* **Use in Base** = current when playing only the base game.
* **Use in PoK** = current when playing with *Prophecy of Kings*.
* **Use in Codex** = current if your group uses the relevant official Codex print-and-play updates.
* **Use in TE** = current in *Thunder’s Edge*.
* **Deprecated** = earlier printing superseded by an official later printing for the same component family in that content environment.

That matches the TI Wiki’s expansion-symbol and omega-label approach and the MCP data model’s explicit “updated components automatically replace older versions” logic. citeturn38view0turn15view0

## How to read the version labels

The TI Wiki’s version-tracking rules are the most explicit single explanation of the notation. It distinguishes **expansion symbols** from **omega symbols**. Expansion symbols identify where a component first appears; omega symbols identify a later updated printing of a pre-existing component. The same page gives **X-89 Bacterial Weapon** as the canonical example of a three-step chain: original Base, **Ω** from Codex I, and **ΩΩ** from Codex IV. It also notes that some updated printed game components do not physically print the omega symbol, but the wiki uses omega notation anyway for consistency. citeturn38view0

In this report, the labels mean:

| Version label | Meaning |
|---|---|
| original | First official printing for that card family |
| revised | Later official printing with wording cleanup but no omega symbol consistently printed |
| Ω | First official replacement version in an omega chain |
| ΩΩ | Second official replacement version in an omega chain |
| replacement | A practical label for cards like PoK Construction or TE Warfare where the later print supersedes play usage in that content environment |

The rows below use shorthand source codes:

| Code | Official source family |
|---|---|
| BG | Base game official rules / support docs |
| PoK | *Prophecy of Kings* rulebook / support docs |
| C1 | Codex I-era official materials, including Alternate Diplomacy / Alternate Technology Cards / Codex I / Codex I Cards |
| C2 | Codex II: *Affinity* / *Affinity Cards* |
| C3 | Codex III: *Vigil* / *Vigil Cards* |
| C4 | Codex IV: *Liberation* / *Liberation Cards* / Codex 4.5 Technology Card Updates |
| TE | *Thunder’s Edge* rulebook / support docs |

This source-code mapping is grounded in the official FFG support page. citeturn11search0turn12search0

#### Technology-family source inventory

For non-revised faction techs, the source assignment is stable:

* **Base faction-specific technology cards** belong to their original Base factions and remain Base-source printings unless they later receive an official omega replacement. citeturn30view1turn38view0
* **PoK faction-specific technology cards** belong to the seven PoK factions and are first printed in *Prophecy of Kings*. citeturn37view1
* **Council Keleres** faction technology begins in **Codex III**. citeturn12search2turn37view2
* **Thunder’s Edge** introduces additional faction technologies for its new factions. I was able to verify the faction families on the TI Wiki and in the MCP repository README, but I could not verify every single TE faction-tech card face directly from official scans in this run, so those should be marked **unspecified** in a strict per-card database until you ingest the TE card PDF or a trusted scan set. citeturn15view0turn37view3

## Ambiguous and disputed cases

The biggest disputed case I found concerns a cluster of updated action cards. The `ti4-mcp` action-card JSON tags **Master Plan Ω**, **Fighter Conscription Ω**, **Blitz Ω**, **Hack Election Ω**, **Solar Flare Ω**, **Counterstroke Ω**, **Ghost Squad Ω**, and **War Machine Ω** as **Codex III** replacements, but the TI Wiki puts them in a dedicated **“Thunder’s Edge Updated”** section and the version-tracking/omega pages compare the original Codex I versions against **TE-icon** updated versions. I therefore recommend storing these as **Thunder’s Edge replacements**, with a “community-source conflict” note attached. citeturn23view0turn24search1turn26view3turn27view0

**Codex II** looks much less important for the card families you asked about than many players assume. Official and community sources both characterize *Affinity* mainly around **Alliance promissory-note reworks**, faction/alliance reference cards, and Council Keleres support material rather than action / agenda / strategy / generic-technology changes. If your site is only tracking action / strategy / tech / political cards, Codex II should exist in your schema, but it will often have **no rows** for those card types. citeturn29search1turn12search13

**I.I.H.Q. Modernization** is another edge case. The TI Wiki explicitly says the Council Keleres faction technology introduced in Codex III is **moved to Breakthrough in Thunder’s Edge**. That means your database should not just track `replaces`; it also needs a field for **component-type migration** so a card can move from “technology card” to some other official component family without losing provenance. citeturn37view2turn15view0

Finally, there is a practical evidence limitation: Fantasy Flight’s support page is excellent for **release provenance**, but not for **systematic per-card face access** for the entire Base and PoK catalog. For many base and PoK rows, the best official evidence is the existence of the relevant support document plus the general rules/component listing, while exact face text comes from the TI Wiki and machine-readable card databases. Those rows are still high-confidence for source-set attribution, but lower-confidence for “official source link per exact card face.” citeturn11search0turn12search0turn14view0turn14view1turn30view0

## Recommended schema for your website

A version-aware TI4 card database should not treat “card name” as the unique key. It should treat each printed form as its own record.

I recommend this schema:

| Field | Type | Purpose |
|---|---|---|
| `card_id` | string | Stable unique ID for this exact printed form |
| `family_id` | string | Groups all versions of the same named card together |
| `name` | string | Printed card name |
| `normalized_name` | string | Search-safe name without Ω symbols or punctuation |
| `card_type` | enum | action / strategy / agenda-law / agenda-directive / tech-basic / tech-faction / tech-unit-upgrade / other |
| `faction` | nullable string | Faction owner if faction-specific |
| `source_set` | enum | base / pok / codex-1 / codex-2 / codex-3 / codex-4 / thunders-edge / twilights-fall / unspecified |
| `version_label` | enum | original / revised / omega / omega-2 / replacement |
| `replaces_card_id` | nullable string | Direct backward link to superseded printing |
| `superseded_by_card_id` | nullable string | Direct forward link |
| `component_type_migrates_to` | nullable string | Needed for cases like I.I.H.Q. Modernization |
| `official_status` | enum | use-in-base / use-in-pok / use-in-codex / use-in-te / deprecated / unspecified |
| `deck_group` | string | action / agenda / strategy / tech-color / faction-tech / etc. |
| `count_in_deck` | integer | Physical copies in the relevant deck |
| `timing_window` | nullable string | For action cards and some abilities |
| `rules_text` | text | Full normalized card text |
| `notes_diff` | text | Human-readable wording changes against prior version |
| `official_source_code` | string | BG / PoK / C1 / C2 / C3 / C4 / TE |
| `official_source_ref` | string | URL or document anchor stored in your backend |
| `community_source_ref` | nullable string | TI Wiki / BGG / MCP / other verification ref |
| `confidence` | enum | official-face / official-release-only / cross-checked-community / disputed |
| `tags` | array | rider / combat / agenda / tech / relic-fragment / wormhole / etc. |

A useful ingestion rule is:

* one **family** = one conceptual card lineage,
* one **record** = one printed or officially revised wording,
* one **status** = “what should a player use in this rules/content environment?”

That model will let your site answer all the questions players actually ask:

* “Is this Base, PoK, or Codex?”
* “Which printing do I use if I own PoK and use Codex?”
* “What did this replace?”
* “What was the old wording?”
* “Is this still current in Thunder’s Edge?”

The official support-page chronology and the TI Wiki’s version-tracking rules are strong enough to support that model cleanly. citeturn11search0turn38view0turn15view0
